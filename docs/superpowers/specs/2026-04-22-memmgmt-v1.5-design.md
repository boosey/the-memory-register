# memmgmt v1.5 — Design & PRD

**Date:** 2026-04-22
**Status:** Approved (brainstorming phase)
**Parent:** `docs/superpowers/specs/2026-04-22-memmgmt-design.md` (v1 — unchanged)
**Next step:** Implementation plan (via `writing-plans` skill)

---

## 1. Problem

v1 shipped with strong unit-test coverage of the core library but **no visual or end-to-end tests**. The five v1 success criteria (design §12) are all user-facing behaviors — "inventory renders in <10s", "click a card to see structured view", "edit → diff preview → save → undo." None of them are currently exercised by any automated test.

For an app whose core tenet is "intuitive, beautiful UX to help manage my Claude Code life," manual-only verification is insufficient. Silent breakage of any UI path is invisible to CI; the first person to notice a broken click handler, an empty React Flow canvas, or a missing Monaco editor is the user, during manual testing.

Additionally, the v1 build surfaced four issues in `docs/superpowers/followups.md` (notably a slug-codec bug that makes projects with dashes in their names show as ghost slugs). These are valuable data points *precisely because* they would have been caught by E2E tests. v1.5 puts the safety net in place.

## 2. Core Tenet

**v1.5 adds tests. It does not fix bugs.**

Tests document the current behavior of v1. If a test fails, the failure is captured in `docs/superpowers/followups.md` as a v2 candidate — not fixed in v1.5. This keeps v1.5 a clean quality-gate increment and prevents "while I'm in here" scope creep from bleeding into test work.

Corollary: **do not refactor production code to make it testable.** The `resolveHomePaths` env-var override (§5.2) is the one and only production change permitted in v1.5 because it is a test-infrastructure concern, not a behavior change.

## 3. Scope — In

1. **Playwright** as the E2E test framework.
2. **Environment-variable override** on `resolveHomePaths()` to let tests point the app at the existing fixture tree at `tests/fixtures/sample-claude-home/` without touching the user's real `~/.claude/`.
3. **E2E test suite** — one test scenario per v1 success criterion (spec §12), plus explicit tests for the four error states in spec §10 (parse failures, dead imports, ghost slugs, concurrent edits).
4. **Build prompt that uses Chrome DevTools MCP** for per-milestone visual sanity checks during the v1.5 build itself (screenshots, console-error scan, a11y audit). MCP invocation is Claude-assisted, not a hermetic CI gate.
5. **Documented followups** — any failing E2E test produces a new entry in `docs/superpowers/followups.md`, not a bug fix.

## 4. Scope — Explicit Non-Goals

- **No bug fixes in v1.5.** Known issues (slug codec, plugin cache walking, unused `contains` edges, etc.) remain open until v2.
- **No component-level tests** (Testing Library). E2E covers user behavior; core-lib unit tests cover logic; component tests are redundant middle ground.
- **No visual regression** (Percy, Chromatic, screenshot diffs). Expensive to maintain, low ROI for v1.5.
- **No performance benchmarks** beyond the "crawl completes in <10s" success criterion, which Playwright can measure as a timing assertion.
- **No refactor of production code** beyond the `resolveHomePaths` env-var override (§5.2).
- **No CI integration** (GitHub Actions, etc.). Local test execution via `pnpm test:e2e` is sufficient for v1.5; CI is a v2+ concern.

## 5. Architecture

### 5.1 Stack Addition

- `@playwright/test` — E2E test runner + assertion library.
- `playwright` — browser binaries (Chromium only in v1.5; Firefox/WebKit are deferred).

Everything else stays as-is.

### 5.2 Test-Home Override

A single, contained change to the one production module that resolves filesystem locations:

**File:** `src/core/paths.ts`
**Change:** extend `resolveHomePaths()` to honor environment variables.

```ts
export function resolveHomePaths(override?: { home?: string; claudeHome?: string }): HomePaths {
  const claudeHome =
    override?.claudeHome
    ?? process.env.MEMMGMT_CLAUDE_HOME
    ?? path.join(override?.home ?? os.homedir(), ".claude");
  // ... existing derivation of projectsDir, pluginsDir, backupsDir, viewPrefsFile
}
```

Derived semantics:

| Env var | Purpose | When set |
|---|---|---|
| `MEMMGMT_CLAUDE_HOME` | Absolute path to the `.claude` directory to crawl. Derives all other paths (projects, plugins, backups, view-prefs). | Test runs only. Never set in production. |
| `MEMMGMT_EXTRA_PROJECTS` | Path-delimiter-separated list of additional project paths to crawl in addition to slug-reversed discovery. | Test runs that need to exercise Project/Local scope against a known fixture. |

Unit-test coverage for `paths.ts` verifies both env vars are respected and that absence falls back to the existing behavior.

No other production files change in v1.5.

### 5.3 Test Fixture

The existing `tests/fixtures/sample-claude-home/` tree is the authoritative fixture for E2E tests. It already contains:

- A valid global CLAUDE.md, settings.json, keybindings, skills/, commands/
- A plugin at `plugins/acme-plugin/` with manifest, skills, commands
- A live slug at `projects/-tmp-fx-proj/` (whose decoded path `/tmp/fx-proj` is expected to be *missing* — produces a ghost slug for testing)
- A dead slug at `projects/-tmp-fx-proj-dead/`
- A `.fixture-project/` tree outside `projects/` (exercised via `MEMMGMT_EXTRA_PROJECTS`) covering Project/Local scope

Additions required for v1.5:

- A CLAUDE.md with a live `@import` (so the import-resolution E2E test has something to assert against).
- A CLAUDE.md with a dead `@import` pointing at a non-existent file (for the dead-import E2E test).

These additions are fixture-only; they do not change production parser behavior.

### 5.4 Playwright Configuration

**File:** `playwright.config.ts` (new, at repo root)

- Single browser: Chromium headless by default; headed via `pnpm test:e2e:headed`.
- `webServer` entry auto-starts `pnpm build && pnpm start` with `MEMMGMT_CLAUDE_HOME` and `MEMMGMT_EXTRA_PROJECTS` set to fixture paths.
- `baseURL` derived from the ephemeral port the launcher picks; see implementation plan for port-passing strategy.
- Test files live under `tests/e2e/*.spec.ts`.
- Results retained: HTML report at `playwright-report/`, traces on first retry.

### 5.5 E2E Test Suite

One `*.spec.ts` file per v1 success criterion (five files). Plus one file for error states (§10).

| File | Asserts |
|---|---|
| `tests/e2e/01-crawl-under-10s.spec.ts` | Cold page-load to fully-rendered inventory in <10s against fixture. |
| `tests/e2e/02-two-clicks-to-context.spec.ts` | From inventory, reach any artifact's scope + content + override status in ≤2 clicks. |
| `tests/e2e/03-edit-save-undo.spec.ts` | Edit a permission rule value → diff preview → save → backup file appears under the fixture's `memmgmt-backups/` → Undo → original value restored on disk. |
| `tests/e2e/04-import-resolution.spec.ts` | CLAUDE.md with `@./imported.md` shows a visible "imports" relationship in the UI; import target is locatable without reading markdown. |
| `tests/e2e/05-first-run.spec.ts` | Fresh server start → browser opens → inventory populated. No config screen, no setup prompt. |
| `tests/e2e/06-error-states.spec.ts` | Ghost-slug panel renders for dead slug; dead-import indicator renders on importing card; parse-failure badge renders for a malformed fixture file. |

### 5.6 Chrome DevTools MCP Gate

Not a hermetic test — a Claude-assisted visual check invoked during the v1.5 build itself via the `chrome-devtools-mcp:chrome-devtools` skill. The v1.5 build prompt requires the building agent to:

1. After each E2E-test file passes, navigate the live app with Chrome DevTools MCP.
2. Take a full-page screenshot of the primary view the new test exercises.
3. Run a quick console-error scan; zero runtime errors expected.
4. Run a Lighthouse accessibility audit; score ≥85 required.
5. File any concerning output to `docs/superpowers/followups.md` (remember: **v1.5 documents, does not fix**).

This adds an ~5-minute sanity pass per E2E file and surfaces visual regressions that raw E2E assertions miss.

## 6. Success Criteria

v1.5 is complete when:

1. `pnpm test:e2e` runs all six spec files. Each either passes cleanly or is documented with an explicit, dated followup entry.
2. `pnpm test` (existing unit tests) still passes with zero regressions.
3. `docs/superpowers/followups.md` contains explicit entries for every failing E2E test, each with reproduction and a proposed v2 fix.
4. Playwright HTML report at `playwright-report/` is reviewable.
5. A clean commit history tags the v1.5 work (prefix `test(e2e):` on commits).

## 7. Error Handling

- **Playwright binary install fails:** fall back to `pnpm playwright install --with-deps chromium` with admin prompt if needed. Document in plan.
- **Port binding fails during webServer startup:** Playwright's `webServer.port` respects `PORT`; reuse the same 5174–5199 range the v1 launcher uses.
- **Test flakes:** retries configured at 2; any test flaking more than 20% gets moved to `@skip-in-ci` annotation with a followup entry.
- **Fixture drift:** if a v2 change to a parser would make an existing fixture invalid, treat that as a fixture update (v2 concern). For v1.5, the fixture is frozen.

## 8. Testing Strategy (meta)

- The tests ARE the deliverable. Their test strategy is §5.5.
- The `resolveHomePaths` env-var change needs a unit test (one new test in `tests/core/paths.test.ts` or equivalent — this file may not exist yet; add it).
- No snapshot testing. No pixel diffs. No Percy.

## 9. Migration / Rollout

v1.5 is additive. No runtime behavior changes. No migration. After `pnpm install` to pull in Playwright, running `pnpm test` and `pnpm test:e2e` should cover the full safety net.

`MEMMGMT_CLAUDE_HOME` env var is only set by the Playwright test harness; production runs of `npx memmgmt` are unaffected.

## 10. Open Questions (Deferred to Implementation)

- Exact approach for passing the server's ephemeral port to Playwright (likely: bind to a fixed test-only port via `PORT=5999` in Playwright's `webServer.env`).
- Whether to include the `.fixture-project/` path via `MEMMGMT_EXTRA_PROJECTS` in all E2E runs or only the tests that need it.
- Whether the new `paths.test.ts` uses Vitest (existing) or Playwright (new). Recommend: Vitest, since it's a pure unit test.
